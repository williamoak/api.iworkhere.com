# ---------------------------------------------------------------------------
# @myDocBlock v2.3 Specification (for importing into ChatGPT or other projects)
# ---------------------------------------------------------------------------
# This standard defines a normalized, strict docblock format suitable for
# Swagger/OpenAPI generation, automated parsing, and consistent documentation.
#
# GENERAL RULES
# - Use standard /** */ docblock format.
# - Never nest multiline comments inside other multiline comments.
# - The following fields must appear in the exact order shown below.
# - @query, @requestExample, @response, and @requires must always contain valid JSON
#   when present (no placeholders, no ellipses, no pseudo-JSON).
# - Code language is never assumed; provide examples only in the language requested.
# - Author tag is always: @author william.r.oak@gmail.com
# - Docblock safety rule: content MUST NOT include the substring "*/" anywhere
#   except the single closing delimiter at the end of the docblock.
#   If a value would contain "*/" (common in globs like "**/*"), escape "/" as "\/"
#   inside strings (example: "src\/**\/*.ts").
#
# REQUIRED FIELD ORDER
# 1. @file
# 2. @external OR @internal     (v2.3: explicit privacy marker)
# 3. @module
# 4. @tag
# 5. @version
# 6. @author
# 7. @path
# 8. @summary
# 9. @description
# 10. @query                   (optional)
# 11. @requestExample
# 12. @response
# 13. @requires
#
# FIELD NOTES
# - @file:
#     File name or logical unit name.
#
# - @external / @internal:
#     Exactly one should be present.
#     - @external: public endpoint file eligible for Swagger export.
#     - @internal: private file; Swagger tooling must ignore it.
#
# - @module:
#     Logical module name used for grouping.
#
# - @tag:
#     Swagger/OpenAPI tag used for endpoint grouping.
#
# - @version:
#     Required. Use exactly: @version x.y.z  (NO colon form).
#     Default:
#       - If no @version is listed, set it to: @version 1.0.0
#     Incrementing rule (project-specific, base-100 rollover):
#       - On any file modification, increment z by 1.
#       - If z becomes 100, set z = 0 and increment y by 1.
#       - If y becomes 100, set y = 0 and increment x by 1.
#     Notes:
#       - x, y, z are non-negative integers.
#       - This is intentionally not strict SemVer (we roll over at 99).
#
# - @path:
#     Required for all files.
#     - For endpoints: HTTP route path, must start with "/" and must NOT include method/hostname.
#       Convention: src/routes/v1/**/<METHOD>.ts → /v1/** (strip the method filename)
#     - For non-endpoints: use tsconfig.json path alias when it matches (e.g. @services/...),
#       otherwise use repo-relative path (e.g. tests/...).
#
# - @summary:
#     Short description (1–2 sentences).
#
# - @description:
#     Longer explanation.
#     - Multi-line allowed
#     - No nested comments
#     - May describe resolution rules, precedence, or semantics
#
# - @query:
#     JSON object declaring supported query parameters.
#     - Keys are query parameter names
#     - Values describe type, required, defaults, and meaning
#     - Declarative only; no behavior implied
#
#     Example:
#     {
#       "all": {
#         "type": "boolean",
#         "required": false,
#         "description": "Return all records"
#       },
#       "complete": {
#         "type": "boolean",
#         "required": false,
#         "description": "Include extended diagnostics"
#       }
#     }
#
# - @requestExample:
#     Full JSON request body example.
#     Use "none" if the endpoint has no request body.
#
# - @response:
#     Full JSON response body example.
#     Must represent a successful response shape.
#
# - @requires:
#     JSON object or array describing dependencies such as:
#     - environment variables
#     - database tables
#     - indexes
#     - external services
#
# Backward compatibility:
# - v2.2 docblocks remain valid
# - @query remains optional
# ---------------------------------------------------------------------------

# ============================================================================
# @myTestingHints — Specification Definition (v1.0)
# ============================================================================
#
# Purpose:
#   @myTestingHints is a structured, comment-based testing guidance block.
#   It documents testing intent, coverage expectations, and behavioral
#   contracts without writing executable test code or binding to a specific
#   test framework.
#
# Placement Rules:
#   - Appears after the top-level documentation block (e.g. @myDocBlock)
#   - May be applied at file, class, or function scope
#   - One @myTestingHints block per logical unit
#
# Structural Rules:
#   - Uses the host language’s comment syntax
#   - Uses tagged sections (not free-form prose)
#   - Each hint must describe a testable behavior
#   - No nested multiline comments
#
# Canonical Sections:
#   @language:
#   @runtime:
#   @harness:
#   @coverage:
#   @entryPoints:
#   @dependencies:
#   @happyPath:
#   @validation:
#   @errorHandling:
#   @edgeCases:
#   @security:
#   @performance:
#   @notes:
#
# Relationship to @myDocBlock:
#   - @myDocBlock defines what the code is and does
#   - @myTestingHints defines how the code should be tested

# ============================================================================
# @myPermanentRules — Working Agreement (v1.0)
# ============================================================================
#
# Scope:
#   - Applies to all work performed in this repository collaboration.
#
# Rules:
#   1) One step at a time:
#      - Change one file (or one tightly-scoped change) per step.
#      - Confirm the step outcome before moving on.
#
#   2) More context in code edits:
#      - Always specify the full file path for any snippet.
#      - For partial snippets, include enough surrounding context to show exactly
#        where it goes in the file.
#
#   3) @myDocBlock required:
#      - Every edited source file must have a top-of-file @myDocBlock.
#
#   4) Public vs private:
#      - Only src/routes/** files are @external.
#      - All other files are @internal.
#
#   5) @path rules:
#      - Endpoints: derive /v1/** from filesystem convention.
#      - Non-endpoints: use tsconfig.json alias when possible; else repo-relative.
#
#   6) @author is fixed:
#      - Always: @author william.r.oak@gmail.com
#
#   7) Docblock terminator safety:
#      - Exactly one "*/" in a @myDocBlock: the closing delimiter only.
#      - Escape "/" as "\/" inside strings as needed to avoid accidental "*/".
#
#   8) Endpoint handler standard:
#      - Express everywhere: route handlers use (req: Request, res: Response).
#
# ============================================================================
